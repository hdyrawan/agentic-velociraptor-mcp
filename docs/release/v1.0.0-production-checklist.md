# v1.0.0 production deployment checklist

The release-level gate for putting v1.0.0 into production. v1.0.0 is
production-ready for **strict controlled deployment** only: read-only
by default, `controlled` mode solely with human out-of-band approvals,
exact-name allowlists, a fail-closed audit log, and least-privilege
Velociraptor identities. Complete every section in order; archive the
completed checklist with your deployment records. All paths are
generic placeholders — substitute your own.

This checklist composes the detailed documents rather than duplicating
them:
[security-review-checklist-v0.10.4.md](../security-review-checklist-v0.10.4.md)
(the security gate),
[production-deployment.md](../production-deployment.md) (hardening),
and the [runbooks](../runbooks/) (operations).

## 1. Pre-deployment

- [ ] Running the released build: `agentic-velociraptor-mcp --version`
      prints `1.0.0`; container image built from the `v1.0.0` tag and
      deployed by pinned tag or digest, never `:latest`.
- [ ] [security-review-checklist-v0.10.4.md](../security-review-checklist-v0.10.4.md)
      completed in full for this environment (tool inventory = 28, no
      raw-VQL/generic-query tool, approval gates, audit fail-closed,
      allowlists, IOC limits, named-source behavior, byte/row limits,
      TLS pinned-server-name, file permissions, secret scan).
- [ ] Velociraptor reader + investigator identities provisioned per
      [velociraptor-permissions.md](../velociraptor-permissions.md);
      neither holds any "never grant" permission.
- [ ] Config built from
      [examples/config/](../../examples/config/) — the deployment
      starts in the **read-only posture**
      (`config.readonly.example.yaml` template) regardless of the end
      goal.
- [ ] Secrets (`*.api.config.yaml`) mode `0600`, outside version
      control, mounted read-only; config change-controlled.
- [ ] Audit storage provisioned with rotation
      (`max_size_bytes`/`max_files`) and retention appropriate for
      investigation records.

## 2. Deployment

- [ ] Deploy per [production-deployment.md](../production-deployment.md):
      non-root container (distroless, no shell), stdio only, **no
      exposed ports**, `--read-only --cap-drop=ALL
      --security-opt no-new-privileges` (or Kubernetes equivalents),
      writable mounts only for `audit.path` (and `download_dir` if and
      when enabled).
- [ ] MCP client wired per
      [mcp-client-integration.md](../mcp-client-integration.md); one
      config per client integration, posture visible per entry.

## 3. Operator acceptance (run against the deployed instance)

- [ ] MCP Inspector smoke checklist passes
      ([mcp-client-integration.md](../mcp-client-integration.md)):
      exactly 28 tools; no tool accepts a query string.
- [ ] `velo_health_check` reports mode `"real"` and a healthy server.
- [ ] A read tool returns real data and respects `limit`.
- [ ] Named-source behavior verified live: `velo_get_flow_results` on
      a `Generic.Client.Info` collection returns
      `status: "source_required"` with real `available_sources`, and a
      retry with `source` set returns rows.
- [ ] Gated-write refusal verified: an approval-gated call with an
      unknown `approval_reference` is refused (`disabled` in read-only
      posture; `not_found` in controlled posture) and nothing reaches
      Velociraptor.
- [ ] If/when switching to controlled posture: one full approval
      round-trip exercised (human `approve` CLI → tool call succeeds →
      approval consumed → mismatched retry rejected), per
      [runbooks/approval-and-audit.md](../runbooks/approval-and-audit.md).
- [ ] Every action above produced exactly one audit event at the
      configured `audit.path`.

## 4. Monitoring and audit operation

- [ ] Daily audit review in place: every `blocked`/`error` event
      investigated; every consumed approval reconciled against its
      case record
      ([runbooks/approval-and-audit.md](../runbooks/approval-and-audit.md)).
- [ ] Audit rotation observed working (or storage headroom monitored);
      rotated files shipped/retained per policy.
- [ ] Alerting (or at minimum scheduled review) on: audit-write
      failures (writes will block — that is correct behavior),
      repeated `blocked` events from one integration, server startup
      failures.
- [ ] Velociraptor server-side logs cross-checkable against this
      server's audit trail (time sync, identity names recorded).

## 5. Emergency disable and rollback

Rehearse before go-live; full procedures in
[runbooks/rollback.md](../runbooks/rollback.md):

- [ ] **Emergency read-only drill completed**: set
      `policy.mode: read_only` (and optionally
      `approval.store_path: ""`), restart, confirm writes refuse and
      reads still work.
- [ ] **Client-access kill drill completed**: remove the MCP client's
      server entry (or stop the client host); confirm no other path to
      the server exists (stdio-only — nothing to firewall).
- [ ] Velociraptor-side identity revocation procedure known to the
      on-call operator (revoke in Velociraptor first; this server has
      no independent revocation).
- [ ] Previous known-good image tag/digest recorded; rollback is a
      redeploy of that pin.

## 6. Known limitations accepted for v1.0.0

The deployment owner explicitly accepts these documented limitations
(details in PROJECT_STATE.md and
[runbooks/controlled-pilot.md](../runbooks/controlled-pilot.md)):

- [ ] Label-scoped hunts, Windows clients, and real file-producing
      uploads/downloads were not exercised live before this release —
      treat each first production use as a validation event (preview
      scope first, verify results against the Velociraptor GUI).
- [ ] Explicit `client_ids` hunt scope is unsupported in real mode
      (no typed RPC); the tools refuse it **before consuming the
      approval** — use `label` scoping.
- [ ] IOC hunting: only `kind: "hash"` performs a real hunt (via
      `Generic.Detection.HashHunter`, which must be allowlisted);
      `ip`/`domain`/`process`/`path` fail closed until curated
      artifacts are installed for them.
- [ ] The `approve` CLI does not enforce requester ≠ approver —
      enforced procedurally.
- [ ] The approval store is a single-operator design; no concurrent
      bulk approvals.
- [ ] Velociraptor-side ACLs remain the primary security boundary;
      this server is defense in depth on top of them, not a substitute.
