## agentic-velociraptor-mcp v1.0.0 — first production release

A secure-by-design MCP server exposing safe, auditable,
policy-controlled Velociraptor endpoint DFIR capabilities to
MCP-compatible agents. v1.0.0 is production-ready for **strict
controlled deployment**: read-only by default; `controlled` mode only
with human out-of-band approvals, exact-name allowlists, a fail-closed
audit log, and separate least-privilege Velociraptor API identities.

### What ships

- **Exactly 28 MCP tools** — 14 read-only (health, client search/info,
  artifact catalog, DFIR profiles, workflow helpers, flow/hunt
  listing and results) and 14 approval/policy-gated (single-client
  collection pilot, flow cancel, uploads/evidence download, hunt
  start/start-DFIR/cancel, fixed-template IOC hunting).
- **stdio transport only.** No listening port, ever. The server runs
  as a child process of an MCP client host.
- **No raw VQL, no generic query execution, no agent-supplied
  artifacts.** Every capability maps to an allowlisted artifact or
  reviewed DFIR profile; the hand-scoped gRPC proto mirror does not
  even declare Velociraptor's generic `Query` RPC or the proto fields
  that carry VQL text. `policy.allow_raw_vql: true` is rejected at
  startup.
- **Human-in-the-loop writes.** No MCP tool can create or decide an
  approval — only the separate operator-run `approve` CLI can. Every
  approval is fingerprinted to the exact operation/target/scope,
  single-use, and TTL-bounded.
- **Fail-closed posture throughout**: read-only default; write pilot
  requires two explicit config switches; audit-write failure blocks
  writes; misconfigured credentials fail startup rather than degrade;
  unsupported preconditions refuse *before* consuming approvals.
- **Typed gRPC bindings for every RPC group**, live-validated against
  a real Velociraptor 0.76.3 lab (v0.10.2), with both live-found bugs
  fixed (v0.10.3): named-source result retrieval (optional `source`
  input, `status: "source_required"` disambiguation) and hash IOC
  hunts via the real, catalog-verified `Generic.Detection.HashHunter`.
- **46 curated DFIR profiles** backed by a reviewed artifact catalog.
- **Operations material** (v0.10.4): production-safe config examples,
  runbooks (approvals & audit, rollback/containment, staged
  first-deployment plan), deployment hardening (non-root distroless
  image, pinning, SBOM/scan guidance), MCP client integration notes,
  and a pre-deployment security checklist.

### Deploying

Start from
[docs/release/v1.0.0-production-checklist.md](v1.0.0-production-checklist.md)
— it sequences the security gate, hardening, operator acceptance,
monitoring, and emergency drills. Begin in the read-only posture
(`examples/config/config.readonly.example.yaml`) and move to
`controlled` (`examples/config/config.controlled.example.yaml`) only
through that checklist. The staged plan in
[docs/runbooks/controlled-pilot.md](../runbooks/controlled-pilot.md)
remains the recommended first-deployment procedure.

### Known limitations (documented, not hidden)

- Label-scoped hunts, Windows clients, and real file-producing
  uploads/downloads were not exercised live pre-release; treat first
  production use of each as a validation event.
- Explicit `client_ids` hunt scope is unsupported in real mode (no
  typed Velociraptor RPC); refused before the approval is consumed —
  use `label` scoping.
- IOC hunting supports `hash` only (`Generic.Detection.HashHunter`);
  `ip`/`domain`/`process`/`path` fail closed until curated artifacts
  are installed for them.
- The `approve` CLI does not enforce requester ≠ approver (procedural
  control); the approval store is a single-operator design.
- This server is defense in depth on top of Velociraptor's own ACLs —
  the least-privilege API identities remain the primary boundary.
